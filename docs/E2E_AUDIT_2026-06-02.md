# End-to-End Engineering Audit

Reviewed: 2026-06-02

## Scope

This review covers the Telegram command surface, SnapTrade onboarding and
webhooks, brokerage synchronization, trade normalization, provisional alerts,
Postgres state, Redis/BullMQ scheduling, deployment health, CI, dependency
security, and open-source secret hygiene.

## Fixed In This Pass

- Added `/reconnect [broker]` using SnapTrade Connection Portal repair mode so
  disabled authorizations are repaired instead of duplicated.
- Preserved SnapTrade webhook retries when Postgres or Redis fails after replay
  protection begins. Only Prisma `P2002` is treated as a completed duplicate.
- Rejected implausibly future-dated SnapTrade webhooks while allowing small
  clock skew.
- Upgraded matching yellow provisional Telegram alerts inline when the
  broker-confirmed execution later arrives.
- Separated the optional realtime `recentOrders` endpoint from the standard
  historical-order watermark. A valid installation no longer suppresses normal
  alerts forever when the optional endpoint is unavailable.
- Reconciled local connection and account rows when SnapTrade stops returning
  them, so stale duplicates no longer remain active in status views.
- Expanded `/healthz` to verify both Postgres and Redis, because both are
  required for alert delivery.
- Converted background timer enqueue failures into deliberate warnings and
  made fatal process errors drain and exit cleanly.

## SnapTrade Capability Map

| Capability | Current use | Recommendation |
| --- | --- | --- |
| Read-only Connection Portal | Used with `connectionType: read` | Keep. This is the right trust boundary. |
| Reconnect mode | Added in this pass | Keep `/reconnect`; do not route disabled connections through normal `/connect`. |
| List connections and accounts | Used and now reconciled locally | Keep. Add richer account sync status to `/diagnostics` later. |
| Standard account orders | Used as the durable execution feed | Keep as the baseline. It may be cached by brokerage. |
| Realtime `recentOrders` | Used as an optional fast path | Ask SnapTrade support to confirm entitlement and brokerage coverage. |
| Unified positions | Used with legacy fallback | Keep. Positions are diagnostic/provisional evidence, not confirmed executions. |
| Manual connection refresh | SDK wrapper exists but is intentionally not called automatically | Add only as an explicit operator or paid-plan feature because refreshes may incur charges and are disabled on realtime plans. |
| `ACCOUNT_HOLDINGS_UPDATED` webhooks | Used to enqueue sync | Also persist `details` success/error fields for better diagnostics. |
| `TRADE_DETECTION` webhooks | Accepted as sync triggers | Ask SnapTrade support to enable account subscriptions where available. This is the best route to seconds-level alerts. |
| `TRADE_UPDATE` webhooks | Accepted as sync triggers | Low value for this read-only product because it tracks trades placed through SnapTrade. |
| Account detail sync metadata | Not used | Add `last_successful_sync`, initial-sync state, and disabled freshness to `/diagnostics`. |
| Historical activities | Not used | Consider a separate daily enrichment job for fees and realized-return quality. Do not use it as the realtime alert source. |
| Rotate user secret | Not exposed | Add an operator runbook before public launch. It is security recovery tooling, not a user command. |

## Remaining Release Priorities

### P1 Before Wider Beta

1. Confirm SnapTrade `recentOrders` and `TRADE_DETECTION` access with SnapTrade
   support for each target brokerage.
2. Persist account-detail sync metadata and webhook `details` errors, then expose
   them through `/diagnostics`.
3. Add an external monitor for Redis readiness now that `/healthz` reports it.
4. Test `/reconnect`, provisional inline upgrade, and Redis outage recovery in
   the Railway staging environment before production deployment.

### P2 Before Public Launch

1. Split API and worker processes when usage approaches 100 connected users.
2. Add a Redis-backed per-user sync lock to avoid redundant SnapTrade calls when
   multiple webhook triggers arrive together.
3. Upgrade `snaptrade-typescript-sdk` from `9.0.200` to the current major version
   on a dedicated compatibility branch.
4. Add a daily activities enrichment job if fee-adjusted sell returns are a
   product requirement.
